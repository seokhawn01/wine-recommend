와인 x 콜키지 매칭 서비스 MVP PRD

1. 핵심 정보

목적: 콜키지 식당을 방문할 때 가져갈 와인에 잘 어울리는 메뉴를 AI 소믈리에가 추천해준다.

사용자: 와인을 즐기지만 메뉴 페어링에 자신이 없는 20~40대 직장인, 콜키지 문화에 관심 있는 소비자.

2. 사용자 여정

핵심 흐름: 와인 취향 설정 → 근처 콜키지 식당 탐색 → 와인/상황 선택 → 메뉴별 페어링 점수 확인 → 메뉴 선택 → AI 소믈리에 설명

1. 앱 최초 실행 (랜딩 페이지)
   └─ 회원가입 클릭
       ↓

2. 회원가입 페이지
   └─ 가입 완료 → 온보딩 페이지로 자동 이동
       ↓

3. 온보딩 페이지 (취향 설정)
   └─ 와인 취향 슬라이더 + 못먹는 음식 선택 → 저장
       ↓

4. 홈 페이지 (주변 콜키지 식당 목록)
   └─ GPS 기반 식당 목록 표시 (메뉴 데이터 있는 식당만 노출)
       ├─ 필터/정렬 적용 → 목록 갱신
       └─ 식당 카드 클릭
           ↓

5. 식당 상세 페이지
   ├─ 와인 선택 (내 와인 컬렉션 또는 취향 프로파일 기본값 사용)
   ├─ 상황 선택 태그 (데이트 / 친구 모임 / 비즈니스 / 혼술 / 가족 모임)
   └─ 메뉴 목록 + 페어링 점수 표시 → 메뉴 선택
       ↓

6. AI 매칭 페이지 (메뉴 선택 후)
   └─ 선택한 메뉴와 와인의 페어링 이유 + 상황 맞춤 소믈리에 설명 표시
       └─ 방문 후 피드백 제출 (좋아요/별로 + 자유 텍스트 후기) → 취향 벡터 자동 업데이트

7. 마이페이지
   ├─ 취향 프로파일 수정
   ├─ 취향 변화 시각화 (레이더 차트 시계열)
   └─ 내 와인 컬렉션 (와인 목록 조회/추가/삭제)

F-18텍스트 후기 기반 취향 학습후기 텍스트 → GPT 분석 → 취향 프로파일 수치 정밀 업데이트AI 매칭

4. 메뉴 구조

[비로그인 상태]
├─ 랜딩 페이지
├─ 로그인
└─ 회원가입

[로그인 후 공통 하단 탭]
├─ 홈 (주변 식당 목록)
├─ AI 매칭 결과
└─ 마이페이지

[홈 → 식당 상세]
└─ 메뉴 데이터 있는 식당만 표시
    └─ 와인 선택 + 상황 선택 → 메뉴 목록 (페어링 점수) → 메뉴 선택 → AI 매칭

[AI 매칭 페이지 내]
└─ 방문 후 피드백 (좋아요/별로 + 텍스트 후기) → 취향 업데이트

[마이페이지 내]
├─ 취향 수정
├─ 취향 변화 시각화
└─ 내 와인 컬렉션 (와인 목록 / 와인 추가 / 라벨 OCR 등록)

7. AI 파이프라인 명세

7.1 와인 라벨 OCR 등록 파이프라인 (F-16)

와인 라벨 사진 촬영
    ↓
Supabase Storage 업로드
    ↓
Google Cloud Vision API
    → OCR 텍스트 추출 (라벨 한글/영문)
    ↓
GPT-4o-mini (Structured Output)
    → 와인명 / 품종 / 빈티지 / 지역 파싱
    → 해당 와인 특성 매핑 (탄닌/산미/바디감/당도 0.0~5.0)
    ↓
wines 테이블 저장

7.2 메뉴 페어링 점수 계산 (식당 상세 페이지 진입 시)

[와인 선택 시] wines 테이블에서 선택 와인 특성값 조회
[미선택 시]   taste_profiles에서 사용자 취향 프로파일 조회 (폴백)
    ↓
못먹는 음식(dietary_restrictions) 해당 메뉴 제외
    ↓
와인 특성값 → 자연어 텍스트 변환
    (예: "탄닌 4.0, 산미 3.5의 풀바디 레드 와인")
    ↓
OpenAI text-embedding-3-small → 쿼리 벡터 생성 (1536차원)
    ↓
pgvector cosine_similarity 검색 → 해당 식당 메뉴 유사도 계산
    ↓
와인-음식 페어링 가중치 적용
    - 탄닌 × 지방성: +2.0  (탄닌이 지방과 결합하여 부드러운 미감)
    - 산미 × 지방성: +1.5  (산미가 기름기를 정리하여 개운함)
    - 탄닌 × 매운맛: -2.5  (둘 다 자극적이어서 충돌)
    - 당도 (와인 >= 음식): +1.0  (와인이 더 달아야 균형)
    - 바디감 × 감칠맛: +1.0  (풍부한 바디가 감칠맛과 조화)
    ↓
메뉴별 최종 페어링 점수 → 식당 상세 메뉴 목록에 ★ 아이콘으로 표시

GPT-4o 입력값:
    - 선택 와인 정보 (품종/지역/탄닌/산미/바디감/당도)
    - 오늘의 상황 (데이트/친구모임/비즈니스/혼술/가족모임)
    - 선택 메뉴 맛 프로파일 (fattiness/umami/spiciness 등)
    - 페어링 점수 근거
    ↓
GPT-4o
    → 상황 맞춤 소믈리에 스타일 페어링 이유 2~3문장 생성
    예: "데이트 자리에 딱 어울리는 선택이에요.
        이 와인의 섬세한 탄닌이 스테이크 지방과 만나 ..."
    ↓
AI 매칭 페이지 렌더링

7.4 텍스트 후기 기반 취향 학습 (피드백 제출 시, F-18)

피드백 제출 (좋아요/별로 + 텍스트 후기 선택 입력)
    ↓
[텍스트 후기 있을 경우]
GPT-4o-mini → 텍스트에서 선호/비선호 요소 추출
    예: "와인이 너무 강했어요" → 탄닌 과다로 해석
    예: "산미가 음식의 기름기를 잡아줘서 좋았어요" → 산미×지방성 조합 선호 강화
    → Structured Output: { tannin: -0.3, acidity: +0.2, body: 0, sweetness: 0 }
    ↓
[텍스트 없을 경우]
    단순 +/- 방향 업데이트
    ↓
taste_profiles 수치 조정 (탄닌/산미/바디감/당도)
feedbacks.profile_adjustments 에 조정값 로그 저장
    ↓
다음 추천부터 조정된 프로파일 반영
모델 파인튜닝 아님. GPT가 텍스트를 분석해 구조화된 수치 조정값을 뽑아내는 방식.

7.5 사용자 피드백 기반 식당 별점 업데이트 (피드백 제출 시)

피드백 제출 (좋아요 또는 별로)
    ↓
restaurants 테이블에서 해당 식당의 naver_rating, feedback_count 조회
    ↓
베이지안 가중 평균 공식으로 새 rating 계산:

    rating = (naver_rating × W + Σ feedback_score) / (W + feedback_count)

    - W (초기 가중치) = 10  ← 피드백 10개 분량의 신뢰도를 네이버 별점에 부여
    - feedback_score: 좋아요 = 5점, 별로 = 1점
    - feedback_count: 해당 식당의 누적 피드백 수

    예시:
      네이버 별점 4.0, 피드백 없음  → rating = 4.0
      피드백 5개 (좋아요 4, 별로 1) → (4.0×10 + 4×5 + 1×1) / (10+5) ≈ 4.07
      피드백 50개 (좋아요 40, 별로 10) → (4.0×10 + 40×5 + 10×1) / (10+50) ≈ 4.17
    ↓
restaurants.rating, restaurants.feedback_count 업데이트
초기 피드백이 적을 때 별점이 과도하게 흔들리는 것을 방지하기 위해 베이지안 방식 사용.
naver_rating은 운영자 수기 입력 시 네이버 플레이스에서 확인한 값으로 한 번만 설정되며 이후 변경되지 않음.

8. 식당 데이터 전략

네이버 Local Search API는 식당 기본 정보(이름, 주소, 좌표, 전화번호, 카테고리)만 제공한다. 콜키지 조건과 메뉴 데이터는 API로 수집할 수 없으므로 아래 전략을 사용한다.

네이버 API 제공 범위

데이터제공 여부
식당명, 주소, 좌표제공
전화번호, 음식 카테고리제공
콜키지 허용 여부미제공
콜키지 비용 / 병 수 제한미제공
메뉴 항목 / 가격미제공
영업시간불안정 (미제공 케이스 多)

초기 데이터 구축 (운영자 직접 입력)

  • 강남/홍대 콜키지 식당 50곳을 네이버지도로 직접 조사
  • 콜키지 허용 여부, 비용, 병 수 제한, 예약 필요 여부 확인 후 Supabase에 수기 입력
  • 식당당 대표 메뉴 5~10개 직접 입력 (별도 admin 페이지 없이 Supabase 대시보드 직접 사용)
  • 메뉴 입력 후 맛 프로파일 벡터화는 백그라운드 스크립트로 처리

홈 노출 기준

  • menus 테이블에 해당 restaurant_id로 1개 이상의 메뉴가 존재하는 식당만 홈에 표시
  • 메뉴 미입력 식당은 DB에 보관하되 유저에게 노출하지 않음

별점 및 콜키지 조건 관리

  • 네이버 별점을 초기값으로 사용, 피드백 누적에 따라 앱 내 별점 진화 (베이지안 가중 평균, Section 7.5 참고)
  • 운영자 수기 입력 시 네이버 플레이스에서 확인한 별점을 naver_rating으로 함께 입력
  • 사용자가 콜키지 조건 변경 신고 → 운영자 확인 후 반영

14. 검증 시나리오

  • 와인 라벨 OCR → wines 테이블 저장 정상 확인
  • 식당 상세에서 와인/상황 선택 후 페어링 점수 변화 확인
  • 와인 미선택 시 취향 프로파일 폴백 동작 확인
  • 텍스트 후기 입력 → GPT 분석 → taste_profiles 수치 변경 확인
  • 데이트 vs 친구모임 상황에 따른 GPT 출력 차이 확인
  • 홈에서 메뉴 없는 식당 미노출 확인
  • 식당 등록 시 naver_rating = 4.2 입력 → rating = 4.2, feedback_count = 0 확인
  • 좋아요 피드백 제출 → rating이 소폭 상승하는지 확인
  • 별로 피드백 다수 제출 → rating이 하락하는지 확인
  • 피드백 0개일 때 rating = naver_rating 동일 확인
  • 홈 페이지 별점순 정렬 시 업데이트된 rating 기준으로 정렬되는지 확인
